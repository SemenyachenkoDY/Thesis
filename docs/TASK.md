# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## Разработка прототипа микросервиса быстрой подписи документов Fast Sign

Основание: ВКР «Разработка прототипа микросервиса быстрой подписи документов для B2B-платформы и оценка его эффективности» Разработка прототипа микросервиса быстрой подписи документов для B2B-платформы и оценка его эффективности

---

# 1. Общие положения

## 1.1. Цель разработки

Создание прототипа микросервиса Fast Sign, обеспечивающего ускоренный, отказоустойчивый и аналитически прозрачный процесс подписания документов в B2B-платформе.

## 1.2. Назначение системы

Микросервис Fast Sign предназначен для:

* обработки операций электронной подписи документов;
* управления жизненным циклом подписи;
* генерации и публикации событий;
* обеспечения интеграции с аналитическим контуром;
* минимизации ручного вмешательства.

---

# 2. Область применения

Сервис используется в составе B2B-платформы (типовой брокерский сценарий), аналогично модели, описанной в дипломе (гипотетическая компания «ИнвестПорт»).

Пользователи:

* Клиенты (юридические лица, физические лица, партнёры);
* CRM-операторы;
* Аналитики;
* Системные администраторы.

---

# 3. Архитектурные требования

## 3.1. Архитектурный стиль

* Микросервисная архитектура
* Event-Driven Architecture (EDA)
* REST API
* Асинхронное взаимодействие через брокер сообщений

## 3.2. Логическая архитектура

Компоненты:

1. Fast Sign Service (основной микросервис)
2. DocService (генерация документа)
3. Message Broker (Kafka/RabbitMQ)
4. PostgreSQL (хранилище)
5. Analytics Service
6. Mock SMS Provider
7. Mock KEP Provider

Взаимодействие строго через API и события.

---

# 4. Функциональные требования

## 4.1. Управление документами

Система должна обеспечивать:

FR-1. Создание документа.
FR-2. Присвоение UUID.
FR-3. Хранение статуса документа.
FR-4. Расчёт времени обработки.
FR-5. Обновление статуса по событию.

Статусы документа:

* CREATED
* SIGNATURE_REQUESTED
* SIGNED
* FAILED
* EXPIRED

---

## 4.2. Поддержка типов подписи

FR-6. Поддержка квалифицированной электронной подписи (КЭП).
FR-7. Поддержка простой электронной подписи (SMS).
FR-8. Поддержка повторных попыток.
FR-9. Ограничение количества попыток (параметризуемое).

---

## 4.3. Событийная модель

Сервис обязан публиковать события:

* DocumentCreated
* SignatureRequested
* SignatureAttempted
* DocumentSigned
* SignatureFailed
* DocumentExpired

Каждое событие должно содержать:

* event_id
* event_type
* entity_id
* timestamp
* payload (JSON)

---

## 4.4. Аналитика

FR-10. Система должна обеспечивать расчёт KPI:

* среднее время подписания;
* коэффициент успешности;
* среднее количество попыток;
* доля ошибок;
* количество обращений в поддержку.

FR-11. Сервис должен предоставлять endpoint для выгрузки агрегированных метрик.

---

NFR-3. Горизонтальное масштабирование через контейнеризацию.

# 7. API Спецификация

## 7.1. Создание документа

POST /documents

Request:

```json
{
  "client_id": "uuid",
  "document_type": "agreement",
  "scenario": "TO_BE"
}
```

Response:

```json
{
  "document_id": "uuid",
  "status": "CREATED"
}
```

---

## 7.2. Инициация подписи

POST /documents/{id}/sign

Request:

```json
{
  "signature_type": "FAST_SIGN"
}
```

Response:

```json
{
  "status": "SIGNATURE_REQUESTED"
}
```

---

## 7.3. Подтверждение SMS

POST /documents/{id}/confirm

---

## 7.4. Получение статуса

GET /documents/{id}

---

## 7.5. Получение KPI

GET /analytics/kpi?period=2025-01

# 8. Сценарии работы (Use Cases)

## UC-1 Подписание через Fast Sign (TO-BE)

1. Клиент создаёт документ.
2. Сервис публикует DocumentCreated.
3. Клиент инициирует подпись.
4. Генерируется SignatureRequested.
5. Подпись выполняется.
6. Генерируется DocumentSigned.
7. Статус обновляется автоматически.

## UC-2 Ошибка подписи

1. Возникает ошибка.
2. Публикуется SignatureFailed.
3. Разрешается повторная попытка.
4. При превышении лимита — статус FAILED.

---

# 9. Метрики эффективности (Acceptance Criteria)

Прототип считается успешным при выполнении:

* Сокращение среднего времени обработки ≥ 5 раз.
* Success Rate ≥ 97% в TO-BE.
* Снижение количества обращений ≥ 30%.
* Корректная публикация 100% событий.
